# Versioning Strategy At LSTV, we follow a structured versioning strategy.
It's inspired by Semantic Versioning (SemVer) principles and uses the classic three-part version number: `MAJOR.MINOR.PATCH-SUFFIX`, but is not quite the same in exact meaning, because Semantic Versioning isn't always ideal for human readability. ### Examples: - `1.0.0` - `2.5.3` - `3.0.0-beta` ## MAJOR Version

We increment our MAJOR version for huge updates that bring significant changes or breaking/restructuring changes, or mark the next evolution of our software. Unlike SemVer it doesn't always mean breaking changes. If you are using an older version of our software and we release a new MAJOR version, you may need to update your code, though we also often provide compatibility layers, and sometimes a major version can be more of a marker.

## MINOR Version

A MINOR version is incremented when we add larger enhancements, fixes, or feature additions in a backward-compatible manner. These releases can include major functionality updates.

## PATCH Version

A PATCH version is incremented when we make backward-compatible changes. Unlike SemVer, we also increment it for medium/pre-test updates as well in addition to optimizations or bug fixes that retain compatibility.

## Release Cycle

We release updates regularly, but there's no fixed schedule. Major updates often come yearly, but not as a rule.

## Support Policy

We provide support for the latest MAJOR version of our software, as well as the previous MAJOR version, for a limited time. In some cases we keep LTS releases for prior MAJOR versions.
